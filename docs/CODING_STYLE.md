# 代码规范（手动维护）

## 1. 目标与范围
- 目标：保证代码可读、可维护、可审查，减少风格噪音和无价值 diff。
- 适用范围：`src/`、`tests/`、`scripts/` 下的 Python 代码。
- 生效方式：**新文件和改动文件强制执行**；未改动文件暂不批量重写。

## 2. Docstring 规范（Google 风格）
- 新增或修改的 Python 文件，注释风格统一使用 Google 风格 docstring。
- 公开 API（模块级公开函数、公开类、公开方法）必须有 docstring。
- 私有函数（如 `_internal_*`）可简化，但逻辑复杂时必须补充说明。

建议结构如下（按需保留）：

```python
def foo(bar: str, baz: int = 0) -> bool:
    """一句话说明函数作用。

    Args:
        bar: 参数说明。
        baz: 参数说明。

    Returns:
        返回值说明。

    Raises:
        ValueError: 异常触发条件说明。
    """
```

## 3. 渐进迁移策略
- 只在“本次改动涉及的文件”内做注释风格迁移。
- 不做全仓一次性注释重写，不提交纯大规模风格噪音改动。
- 若某文件改动范围很小，仅需迁移受影响函数；无需强制清理全文件全部历史注释。

## 4. 文件头策略（重要）
- 不再手工维护文件头中的 `@Update` 日期。
- 作者、修改时间、变更历史以 Git 记录为准。
- 新文件不再新增 `@Author/@Update` 式头注释，优先使用模块 docstring 说明用途。
- 旧文件的历史头注释可暂时保留；除非本次改动涉及该处，不做专门“改日期”提交。

## 5. 提交建议
- 风格迁移与功能变更尽量分提交，便于审查与回滚。
- 若同一文件同时有功能与注释调整，优先保证行为变更清晰可审查。
